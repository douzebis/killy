# Offline nixos-install â€” research notes

## Problem

`nixos-install --flake` always calls `nix build --store /mnt --flake ...` to
produce the system toplevel. When the target system's `hardware-configuration.nix`
contains real disk UUIDs (unknown at ISO build time), the resulting toplevel has
a different store path than any pre-built version â€” so Nix cannot reuse it
directly and must build it.

The question: how do others achieve fully offline NixOS installation?

---

## Key finding: `auto?trusted=1`

`nixos-install` automatically adds `--extra-substituters auto?trusted=1` to its
internal `nix build` call. This makes the live ISO's running Nix store available
as a trusted substituter for the target store (`/mnt`). Every package path that
exists on the ISO is served locally â€” no network involved. Only the thin toplevel
derivation itself (activation scripts, fstab symlinks â€” a few seconds of work)
actually needs to be built.

**Consequence**: `--option substituters ''` disables `auto?trusted=1` along with
everything else. Without it, Nix cannot find any of the package inputs for the
thin toplevel build and falls back to building the entire bootstrap chain from
scratch â€” hours, not seconds.

Source: [nixos-install source (nixpkgs)](https://github.com/NixOS/nixpkgs/blob/master/nixos/modules/installer/tools/nixos-install.sh)

---

## How the live ISO store gets registered

The squashfs that forms the ISO's read-only root contains:

1. All store paths from the embedded closure (e.g. the killy system closure via
   `isoImage.storeContents`).
2. A `nix-path-registration` file at the squashfs root, generated by
   `make-squashfs.nix` from `closureInfo/registration`.

At boot, `iso-image.nix` runs:

```bash
nix-store --load-db < /nix/store/nix-path-registration
```

This populates the live Nix daemon's DB with all squashfs store paths, making
them valid. This is what `auto?trusted=1` queries when serving paths to
`nix build --store /mnt`.

Sources:
- [make-squashfs.nix](https://github.com/NixOS/nixpkgs/blob/master/nixos/lib/make-squashfs.nix)
- [iso-image.nix](https://github.com/NixOS/nixpkgs/blob/master/nixos/modules/installer/cd-dvd/iso-image.nix)

## The `nixos-anywhere` pattern: `--system` + `nix copy`

`nixos-anywhere` is the reference implementation for automated NixOS deployment.
Its install flow:

```bash
# 1. Copy pre-built closure into /mnt's Nix store (handles DB registration).
nix copy --no-check-sigs --to "local?root=/mnt" "$system_toplevel"

# 2. Install the pre-built path directly â€” skips nix build entirely.
nixos-install --root /mnt --no-root-passwd --no-channel-copy \
    --system "$system_toplevel"
```

With `--system PATH`, `nixos-install` skips the entire `nix build` block and
goes straight to:

```bash
nix-env --store /mnt -p /mnt/nix/var/nix/profiles/system --set "$system_toplevel"
nixos-enter --root /mnt -c "switch-to-configuration boot"
```

No re-evaluation. No build. The installed system is exactly what was pre-built.

`nix copy --to local?root=/mnt` initializes `/mnt/nix/var/nix/db` automatically
and registers the full closure. `nix copy` does not copy `.drv` files â€” only
output paths â€” which is fine because `--system` never needs to evaluate
derivations.

Sources:
- [nixos-anywhere source](https://github.com/nix-community/nixos-anywhere/blob/main/src/nixos-anywhere.sh)
- [local store docs (Nix 2.28)](https://nix.dev/manual/nix/2.28/store/types/local-store)
- [NixOS Discourse: air-gap installer](https://discourse.nixos.org/t/an-airgap-nixos-installer-with-batteries-included/50618)

## The UUID problem with `--system`

The pre-built toplevel contains an fstab generated from whatever
`hardware-configuration.nix` was used at build time. Real disk UUIDs are unknown
at ISO build time, so the pre-built fstab is wrong.

**Solution: mount by filesystem label instead of UUID.**

`killy-install` already formats partitions with fixed labels (`mkfs.btrfs -L
nixos`, `mkfs.fat -n BOOT`). If `hardware.nix` declares `fileSystems` using
`/dev/disk/by-label/...`, the pre-built fstab is correct for any installation of
killy regardless of actual partition UUIDs.

`nixos-generate-config` still runs (for the initrd kernel module list), but its
UUID-based `fileSystems` entries are superseded by the label-based declarations
in `hardware.nix`.

## Approach chosen for killy (spec 0010)

Use `nixos-install --flake` and let `auto?trusted=1` do the work:

```
nixos-install --root /mnt --no-root-passwd --impure --no-channel-copy \
  --flake /mnt/etc/nixos#killy
```

No pre-copy step. `nixos-install` adds `auto?trusted=1` automatically, which
serves all package paths from the live ISO store (registered via
`nix-path-registration` at boot). The flake is re-evaluated against the real
`hardware-configuration.nix` (correct partition UUIDs). Only the thin toplevel
derivation is built â€” seconds, fully local, no network.

`--option substituters ''` must NOT be passed â€” it disables `auto?trusted=1`
along with everything else, forcing Nix to build the entire bootstrap chain
from scratch.
